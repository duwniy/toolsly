# Authentication & Authorization Flow

## Overview
Toolsly uses a stateless JWT authentication model. Users authenticate via email/password, and subsequent requests are authorized using a Bearer token.

## JWT Structure
The JWT generated by `JwtService` contains standard claims plus custom organizational context:
- `sub`: User email
- `user_id`: DB primary key (UUID)
- `role`: One of `ADMIN`, `STAFF`, `RENTER`
- `branch_id`: UUID of the assigned branch (can be null for ADMIN/HR)

## Authentication Endpoint
`POST /api/auth/login`
- **Request**: `email`, `password`
- **Response**: 
  - `token`: JWT string
  - `userId`, `email`, `role`, `branchId`

## Authorization Layer
Implemented via `SecurityConfig` (Spring Security):
- `/api/auth/**`: Public
- `/api/inventory/**`: Staff/Admin
- `/api/orders/issue`: Staff
- `/api/reporting/**`: HR/Admin (with Staff-level branch insulation)

## Branch Insulation
Branch data is isolated at two levels:
1. **API Parameterization**: Controllers accept `branchId` (from the token or explicit param).
2. **Backend Logic**: Repositories and Services (e.g., `InventoryService`, `ReportingService`) use `branchId` to filter SQL results, ensuring a user from Branch A cannot see items from Branch B.
