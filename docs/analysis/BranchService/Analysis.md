# Analysis: Branch Capacity & Storage Rules

## 1. Концепция
Каждый филиал (`Branch`) имеет фиксированный параметр `storage_capacity`. Это физическое ограничение на количество предметов, которые могут находиться на балансе филиала одновременно. 

## 2. Бизнес-правило (BRD)
> "Система не должна позволять завершить возврат инструмента в филиал, если текущее количество предметов в этом филиале равно или превышает его `storage_capacity`."

## 3. Алгоритм проверки (Storage Guard)
При вызове операции `/return`:
1. Найти филиал по `branchId`.
2. Подсчитать количество активных предметов в филиале:
   `SELECT COUNT(*) FROM equipment_items WHERE branch_id = :branchId`
3. Сравнить `currentCount` с `branch.storageCapacity`.
4. Если `currentCount >= storageCapacity`:
   - Выбросить бизнес-исключение `StorageFullException`.
   - API возвращает `422 Unprocessable Entity` (или `409 Conflict`).
5. Если место есть:
   - Обновить статус заказа.
   - Обновить филиал и состояние предмета.

## 4. Оптимизация
Для высоконагруженных систем можно использовать поле `current_occupancy` в таблице `branches` с атомарным инкрементом, но на текущем этапе достаточно простого `COUNT(*)` для обеспечения консистентности.
