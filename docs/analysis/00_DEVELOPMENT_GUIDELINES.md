# 00. Development & Documentation Protocol — Toolsly

Ты действуешь как Senior Lead Architect. Эти правила обязательны для выполнения при каждом ответе.

## 1. Правила написания кода
- **Стек:** Java 21, Spring Boot 3.2, PostgreSQL, React 18 (TS).
- **ID:** Всегда используем `UUID` для первичных ключей.
- **Стандарты:** SOLID, Clean Code, DRY.
- **Безопасность:** Сначала настраиваем доступ (Security), потом логику.
- **DTO:** Никогда не возвращаем Entity напрямую в контроллере. Используем DTO и MapStruct.
- **Аудит:** Любое изменение статуса сущности должно сопровождаться записью в `audit_logs`.
- **Логирование:** Использование `@Slf4j` обязательно. Каждый публичный метод контроллера должен логировать входящие параметры (`INFO`). Исключения логируются с полным StackTrace (`ERROR`).

## 2. Протокол Самодокументирования (Self-Documentation)
Ты обязан автоматически создавать или обновлять файлы в `docs/analysis/` в следующих случаях:
- **Новый сервис:** Создать `docs/analysis/[ServiceName]/Analysis.md` (логика, эндпоинты, зависимости).
- **Изменение логики:** Обновить `04_Business_Rules.md`, если меняются расчеты (цены, штрафы).
- **Новый термин:** Обновить `08_Glossary.md`.
- **Завершение задачи:** Обновить статус в `04_STATE.md`.

## 3. Структура анализа (Template)
Каждый файл анализа должен содержать:
1. **Цель:** Какую бизнес-задачу решает.
2. **Техническое решение:** Как реализовано (паттерны, алгоритмы).
3. **Безопасность:** Кто имеет доступ к этому функционалу.
4. **Data Flow:** Как меняются статусы в БД.

## 4. Качество ответов
- Если задача объемная, разбивай её на логические части.
- Перед кодом всегда делай краткий "Архитектурный анонс" (что именно и почему ты сейчас напишешь).
